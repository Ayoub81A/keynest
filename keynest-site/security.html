Security

KeyNest uses a zero-knowledge architecture. Your encryption key is derived from your password in your browser and never transmitted to or stored by the server.

Encryption

How your data is protected

Every vault entry goes through a multi-layer encryption process entirely inside your browser before a single byte reaches our servers.

1
You enter your master password
Your password is used as the seed for key derivation. It is never stored, transmitted, or logged anywhere.
2
PBKDF2 derives your encryption key
Using a unique per-account salt stored on the server, PBKDF2-SHA256 runs 310,000 iterations to produce a 256-bit encryption key. This makes brute-force attacks computationally expensive.
PBKDF2(password, salt, 310000, SHA-256) → 256-bit key
3
AES-256-GCM encrypts each entry
Each vault entry is serialised to JSON and encrypted with AES-256-GCM. A unique 96-bit IV is generated per operation. The authentication tag ensures tamper detection.
AES-256-GCM(key, iv, plaintext) → ciphertext + authTag
4
Ciphertext sent to server
Only the encrypted blob is transmitted — the key never leaves your browser. The server stores base64-encoded ciphertext it cannot decode.
5
Decryption happens locally on retrieval
When you load your vault, the server sends ciphertext back. Your browser decrypts it using the key in memory. On logout, the key is wiped.
What the server stores
// Entry in database — unreadable

gYW5kIHRoZSBzZXJ2ZXIgY2Fu
bm90IHJlYWQgaXQ="
What only you can read
// Decrypted in browser — never sent
"site": "GitHub",
"username": "alex@example.com",
"password": "Xk#9mP2vQr!"
Zero-knowledge guarantee

Even if our database were compromised, attackers would only obtain ciphertext they cannot decrypt without your master password.

Architecture

How the system is built

A clean separation between what the client handles and what the server handles.

Browser (Client)
HTML + CSS + Vanilla JS
  • ✓Derives encryption key from password using PBKDF2
  • ✓Encrypts entries with AES-256-GCM before sending
  • ✓Decrypts entries after fetching from API
  • ✓Holds encryption key in memory only
  • ✓Wipes key from memory on logout
Server (PHP + MySQL)
PHP 8 + MySQL / XAMPP
  • ✓Handles user registration and authentication
  • ✓Stores bcrypt-hashed passwords (cost 12)
  • ✓Stores per-user PBKDF2 salt (public — not secret)
  • ✓Stores and serves encrypted ciphertext blobs
  • ✗Never sees or stores the encryption key
Tech Specs

Implementation details

Component Specification
Symmetric encryption AES-256-GCM 256-bit key, 96-bit IV (random per operation), 128-bit authentication tag
Key derivation PBKDF2-SHA256 310,000 iterations — meets NIST SP 800-132 recommendations for 2024
KDF salt 32 random bytes (64 hex chars), unique per user, generated server-side with random_bytes()
Password hashing bcrypt PHP password_hash() with cost factor 12
Crypto implementation Browser-native Web Crypto API — no external libraries, no npm dependencies
Session management PHP $_SESSION server-side sessions. Session expires on tab/browser close (sessionStorage flag).
Ciphertext format base64( IV[12 bytes] + AES-GCM-ciphertext + authTag[16 bytes] )
Transport HTTPS recommended in production. All API calls use credentials: 'same-origin'.
Key lifetime In-memory only. Never stored in localStorage, sessionStorage, cookies, or transmitted to server.

FAQ

Common questions

What happens if I forget my master password?
Your vault cannot be recovered without your master password. Since we never store or know your encryption key, there is no password reset mechanism — this is intentional. If you forget your password, your vault entries are permanently unrecoverable. Choose a strong password you won't forget, and consider storing it somewhere very safe.
Can KeyNest employees read my passwords?
No. The server only stores ciphertext encrypted with a key we never see. Even with full database access, entries are mathematically unreadable without the user's master password. This is the zero-knowledge guarantee.
Why PBKDF2 and not Argon2?
Argon2 is theoretically stronger for key derivation because it is memory-hard. However, it is not natively supported in browser Web Crypto APIs and requires a third-party JavaScript library. PBKDF2 is built into every modern browser, requires zero external dependencies, and at 310,000 iterations is more than sufficient for a personal password manager. Argon2 support may be added in a future version.
Is KeyNest safe to use on a shared computer?
Sessions are stored in sessionStorage, so they expire automatically when you close the tab. However, on a shared computer we recommend always clicking Sign Out explicitly, and being cautious about keyloggers or screen recording software that could capture your master password at login.
What happens if the database is breached?
An attacker who obtains the database gets encrypted blobs and bcrypt-hashed passwords. The blobs are unreadable without the per-user encryption key, which is never stored. The bcrypt hashes are designed to be slow to crack. As long as your master password is strong, a database breach does not expose your vault contents.
Is this project open source?
Yes. The full source code is available on GitHub under the MIT license. You can audit the encryption implementation yourself, run your own instance, or contribute improvements.
